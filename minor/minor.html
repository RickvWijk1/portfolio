Arjan reminded us that the IT Cyclus/design thinking is flexible and unlike the waterfall method you can skip a design phase temporarily and return to it later on when you have new insights/knowledge.

Brainstorming

I started looking for ways to get the ideation process going. I suggested the 'Lotus Blossum' method which is recommended by the Digital Society School.The idea is that you start with a central theme and from that theme, eight conceptual themes flow out of the central theme. Each of them are used as central theme to generate 8 more themes and so on. You use this when there is a need to generate a big number of ideas quickly. It's similar to mindmapping or making word webs. I used this method during my internship with success.

We ended up not using this method, but simply started making a word web with Zenya as the central theme. After we wrote some words down it felt like making a word web didn't give us any inspiration/ideas.

Design Sprint method

Ruben found another method that's used to help ideating. It's usually used across a whole week and this process gets repeated multiple times during a project. In our case, we cut it short to an average of two days each time this method was used.

This is how we used it:

1

Write down the main question. What do you want to achieve with this project?

2

Each member writes down as many HMW/HCW questions on sticky notes as they can. These questions help break down the main question.

3

Each member gets a couple of votes to give to specific HMW/HCW (How Might We, How Can We) questions that they think are the most important.

4

Define the routes/steps the user has to take until the user reaches their goal. This is a form of User Journey Mapping.

5

Order the HMW/HCW questions under the user routes/steps where the project team thinks they belong. The questions with the least votes get stored away and used if they're important in future sprints.

6

Come up with arguments on why this project can't succeed. Translate them into research questions.

7

Some of the questions need to be answered to find a possible solution. Each member creates a storyboard that holds a possible solution to the user's problem.

Initial concept

At the start of the hybrid app project a few concepts were already defined by the teachers and put on a Powerpoint presentation. I was interested in the following concept:

Visitors to a pop concert can (if allowed by the organisation of the concert) stream live video (and music) to a server with this app. Users can then log on to that server and follow the concert live on screen.

I was interested in this concept, because I had never done something that involved livestreaming technology before. I also saw that this concept offered a complicated design challenge since there's three different kind of stakeholders involved (visitors at concert, viewers at home, organizers of concerts). As ICT & Media student this convinced me to take up this challenge.

Final concept

After the market research was done, I redefined the concept since the conclusion of the market research was that there is a niche in the market for livestreaming apps that focus on festivals instead of specific artists or concerts.

The final concept:

Visitors to a festival can (if allowed by the organiser of the festival) stream live video (and music) to a server with this app. Users can then log on to that server and follow the concerts of the festival live on screen.

Motivation

Designs are necessary since it can be used for testing in order to improve the app. This type of iterating is a lot quicker than first developing and then adjusting the app. Designs also make sure everything is clear for the developers so that development takes considerably less time.

After the target audience was defined and the user stories were created, I started designing in Adobe XD. I chose Adobe XD since I have more experience with it compared to other design tools like Figma. Figma is generally a better tool for working together with someone else, but since Marco was already working on the livestreaming SDK, I decided to design solo in Adobe XD.

Low-fidelity

I started designing low-fidelity since I was mostly looking what elements I needed to put in and where. In short, getting the right elements and alignment was the main goal of this version whereas colors, images, shapes, etc. didn't matter as much yet. As you can see in the image below, I designed different views/flows for the different types of users. This brought more complications as navigation and content differed depending on the type of user you are. For example, organizers need a separate view to promote their festivals while streamers need to choose a festival and stream it while viewers only want to watch.


The low-fidelity designs in Adobe XD

Medium/high-fidelity

The main goal for these designs was to give the elements the right colors, images and shapes to make it a high quality design. I didn't make the prototype clickable since there was too little time left to test it and thus no reason to make it clickable. In that aspect this is medium-fidelity, but with a high-fidelity look. In this version I made the design in a way that viewers and streamers have the same views available. You can compare it to the way YouTube made their design where all users have access to the 'Go Live' button. This decision was based on the personas. A user might not be able to go to a festival and because of that will become a viewer on the app. However, if they can go they can become a streamer. Organizers still have some separate views.

Dev environment & tools

I used Visual Studio Code over Android Studio since I am more used to it and I like to have everything in one place. I installed some extensions to help me program more efficiently.

  • Awesome Flutter Snippets; a collection of commonly used Flutter classes and methods
  • Flutter Preview UI Guides; is part of the Flutter extension that's mandatory to install for Flutter development. There's a setting that allows you to see the relationship between a parent widget and it's child more easily through visual guidelines.
  • Flutter refactor; another feature of the Flutter extension that allows you to refactor your code. You can, for example, easily wrap another widget around a widget.

Stateful & Stateless

In Flutter a widget is either stateful or stateless. State is (just like in React-Native) used to make something dynamic. A stateless widget never changes a.k.a. static. When the user clicks on a notification bell icon, the bell gets a different color. It requires state.

SDK's & libraries

Whilest searching for any libraries/sdk's that could help me, I found a bunch of third party SDK's that allowed you to set up indoor tracking, mainly meant for indoor navigation functionality (Navigine for example). Most of the SDK's made use of AP's (Access Points) and/or bluetooth beacons and required you to upload a floor plan of the area you want the tracking to happen.

I explored some of these SDK's, but I found out the SDK's were focussed on marketing solutions so did not have functionality to offer for what I was looking for.

After a lot of digging, I found the Android Beacon Library. This library allows you to interact with Beacons. The library shows how to send notifications to users when they are nearby beacons.

I tried to borrow some bluetooth beacons and access points from the ISSD, but sadly they didn't have any available. Ruud gave me a good tip by mentioning that there are apps that allow you to simulate your phone as a beacon. Unfortunately, my phone didn't support this, but I was able to borrow a Google Pixel that did support this.

The beacon library provided a function that checked whether the device was within range of a beacon or not. Events could be triggered like the library showed by adding a function for sending notifications.

val centralMonitoringObserver = Observer { state ->
    if (state == MonitorNotifier.OUTSIDE) {
        Log.d(TAG, "outside beacon region: "+region)
    }
    else {
        Log.d(TAG, "inside beacon region: "+region)
        sendNotification()
        sendAudioFile()
    }
}

I made my own function 'sendAudiofile' to play an audio file as soon as the device of the user is within range of a beacon.

private fun sendAudioFile(){
    val mp: MediaPlayer = MediaPlayer.create(this, R.raw.uclfinalcommentary)
    mp.start()
    mp.isLooping = true
}

I tried to find out how to target specific beacons instead of 'any' beacon so that I could play specific audio files for each individual beacon. Sadly the library didn't have any support/documentation for this and I couldn't find more information about this during my desktop research.

Video player

Since I was stuck on my beacon research, I was looking to implement other features. In my first iteration for the club museum concept I planned to give users the option to watch a video of a historic club moment. My design challenge however, was about making the interaction as immersive as possible. Users should be able to walk around the museum and listen to the historic moments instead of looking at their phone's screen. Despite this I decided to implement a video player, because the target audience (kids) might want to see what happened. They might want to see a beautiful goal being scored. I also wanted to implement a video player so that I could learn a bit more about Android.

var mp: MediaPlayer? = null

val mediaController = MediaController(this)

val offlineUri = Uri.parse("android.resource://$packageName/${R.raw.video}")

videoView.start()
videoView.setOnPreparedListener(OnPreparedListener { mp -> mp.setVolume(0f, 0f) })

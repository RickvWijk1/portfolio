Click counter

A simple React-Native click counter

1

2

I followed the 'learn the basics' documentation of React-Native which first gives an example of importing components and making a View with a simple Text component with 'Hello World' as the string.

import React from 'react';
import { Text, View } from 'react-native';

const HelloWorldApp = () => {
  return (
    <View
      style={{
        flex: 1,
        justifyContent: "center",
        alignItems: "center"
        }}>
        <Text>Hello, world!</Text>
    </View>
  )
}
export default HelloWorldApp;

3

The documentation explained about using state in React-Native which allows components to have their output changed. A click counter is an example where state needs to be applied since the output has to be changed.

return (
    <View style={styles.container}>
        <Text>You clicked {count} times</Text>
            <Button
                onPress={() => setCount(count + 1)}
                title="Click POC"/>
    </View> 
);

Concept & Analysis

Research strategies

Desktop

Field

What have I learned?

  • Learned that more and more foreigner visit Dutch festivals

Initial concept

At the start of the hybrid app project a few concepts were already defined by the teachers and put on a Powerpoint presentation. I was interested in the following concept:

Visitors to a pop concert can (if allowed by the organisation of the concert) stream live video (and music) to a server with this app. Users can then log on to that server and follow the concert live on screen.

I was interested in this concept, because I had never done something that involved livestreaming technology before. I also saw that this concept offered a complicated design challenge since there's three different kind of stakeholders involved (visitors at concert, viewers at home, organizers of concerts). As ICT & Media student this convinced me to take up this challenge.

Final concept

After the market research was done, I redefined the concept since the conclusion of the market research was that there is a niche in the market for livestreaming apps that focus on festivals instead of specific artists or concerts.

The final concept:

Visitors to a festival can (if allowed by the organiser of the festival) stream live video (and music) to a server with this app. Users can then log on to that server and follow the concerts of the festival live on screen.

Evaluation

Now that the problem has been defined and a general solution has been noted, the target audience has to become more concrete by defining it further; who exactly are affected by the problem? Who are the stakeholders that need to get involved in order to solve the problem?

By defining the target audience and creating personas you get a better view on/overview of the target audience. This helps later on in the design proces when user stories have to be generated.

Defining the target audience

My contribution to this research:

  1. Asked (5) people in my environment if they have ever watched concerts live on anything (smartphone, tablet, desktop, smart TV (with internet)) but live TV. Most who did, said they have watched livestreams on their TV by either casting it from their phone to the TV or simply watching it directly from their smart TV.
  2. Defined the target audience by noting the semantics of the target audience.
  3. Created personas based on the semantics. Note! I myself occasionally watch livestreams by casting them to my TV or watching them via a browser. I consider myself part of the target audience and since there was little time to do extensive research I have taken my personal experience and assumptions into account during the creation of the personas.

What have I learned?

  • Learned how to apply glassmorphism; when to use and when not to use and how to combine it with other elements correctly
  • Learned how to create glassmorphism in Adobe XD

Motivation

Designs are necessary since it can be used for testing in order to improve the app. This type of iterating is a lot quicker than first developing and then adjusting the app. Designs also make sure everything is clear for the developers so that development takes considerably less time.

After the target audience was defined and the user stories were created, I started designing in Adobe XD. I chose Adobe XD since I have more experience with it compared to other design tools like Figma. Figma is generally a better tool for working together with someone else, but since Marco was already working on the livestreaming SDK, I decided to design solo in Adobe XD.

Low-fidelity

I started designing low-fidelity since I was mostly looking what elements I needed to put in and where. In short, getting the right elements and alignment was the main goal of this version whereas colors, images, shapes, etc. didn't matter as much yet. As you can see in the image below, I designed different views/flows for the different types of users. This brought more complications as navigation and content differed depending on the type of user you are. For example, organizers need a separate view to promote their festivals while streamers need to choose a festival and stream it while viewers only want to watch.


The low-fidelity designs in Adobe XD

Medium/high-fidelity

The main goal for these designs was to give the elements the right colors, images and shapes to make it a high quality design. I didn't make the prototype clickable since there was too little time left to test it and thus no reason to make it clickable. In that aspect this is medium-fidelity, but with a high-fidelity look. In this version I made the design in a way that viewers and streamers have the same views available. You can compare it to the way YouTube made their design where all users have access to the 'Go Live' button. This decision was based on the personas. A user might not be able to go to a festival and because of that will become a viewer on the app. However, if they can go they can become a streamer. Organizers still have some separate views.

I would have liked to iterate more by testing with users, but as this was a relatively short project I decided it was time to move on to development.

Dev environment & tools

I used Visual Studio Code over Android Studio since I am more used to it and I like to have everything in one place. I installed some extensions to help me program more efficiently.

  • Pubspec Assist; allows you to add dependencies for widgets to the pubspec.yaml within VScode. This means you don't have to look up the version and copy paste that.
  • Awesome Flutter Snippets; a collection of commonly used Flutter classes and methods
  • Flutter Preview UI Guides; is part of the Flutter extension that's mandatory to install for Flutter development. There's a setting that allows you to see the relationship between a parent widget and it's child more easily through visual guidelines.
  • Flutter refactor; another feature of the Flutter extension that allows you to refactor your code. You can, for example, easily wrap another widget around a widget.

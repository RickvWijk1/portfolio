Module


De student heeft een applicatie gerealiseerd met een module pattern. Externe libraries zijn gebruikt om de applicatie te ontwerpen. De applicatie is tot stand gekomen via user stories en een schets van de architectuur.

Ik probeer dit leerdoel aan te tonen door mijn eigen module te ontwikkelen.




Aanleiding

Ja dit is een leerdoel dat ik verplicht moet aantonen, maar dat is niet de enige reden waarom ik onderzoek heb gedaan naar module patterns en een module pattern heb ontwikkeld. Nadat ik in semester 2 de basis van Javascript begreep, begon ik me af te vragen hoe ik mijn functies kon hergebruiken zonder dat ik steeds variabelen moest hernoemen en mijn code daardoor een grote bende werd. Dat heb ik dit semester onderzocht.



Onderzoek

Mijn eerste kennismaking met module patterns was tijdens een DEV workshop. Ik kon niet goed volgen hoe het werkte, maar wat ik wel begreep was dat je met een module pattern je code als het ware zo ontwerpt dat je het makkelijk kan hergebruiken en delen met anderen. Vervolgens heb ik op het internet naar artikelen/posts/tutorials gezocht waarin wordt uitgelegd hoe je een module pattern maakt. Na heel wat browsen kwam ik erachter dat je (om je code modulair te maken) moet bepalen welke onderdelen je van je code toegankelijk wilt maken voor de gebruiker om te kunnen bewerken en welke stukken niet. Dit kan door een private scope te maken en met 'return' kun je vervolgens onderdelen van je code in de global scope zetten.



Uitvoering

Allereerst heb ik vastgesteld wat voor een functie ik modulair wilde maken. Voor de SolidSight website is het handig voor gebruikers om door middel van een zoekbalk bepaalde items te kunnen vinden. Hieronder staat de functie die ik heb gemaakt. 



//this function filters categories (using the search bar)
function searchFunction() {
    //initializing variables
    var input, filter, allcontainer, gridItem, h4, i;
    //declaring variables
    input = document.getElementById('myInput');
    //set the input of the search bar to capital letters 
    filter = input.value.toUpperCase();
    allcontainer = document.getElementById('allcontainer');
    gridItem = allcontainer.getElementsByClassName('location-image');

    //keep running loop if 'i' is less than the grid item's length
    for (i = 0; i < gridItem.length; i++) {
        h4 = gridItem[i].getElementsByTagName('h4')[0];

        //the text in h4 is set to uppercase in order too match the input value of filter (a.k.a. the search bar) which is also set to uppercase
        //if the h4 text contains the value that is in the search bar... else...
        if (h4.innerHTML.toUpperCase().indexOf(filter) > -1) {
            //item is displayed
            gridItem[i].style.display = "";
        } else {
            //item is not displayed
            gridItem[i].style.display = 'none';
        }
    }
}

Om kort samen te vatten wat het doet: als een gebruiker typt in de zoekbalk, wordt er gekeken of de tekst input matched met de 'h4' van een of meerdere grid items.


Vervolgens ben ik gaan kijken hoe ik dit overdraagbaar kan maken. De eerste vraag die ik mezelf stelde was: "Tot welke variabelen in mijn functie mag de gebruiker van mijn code toegang hebben en welke niet?" Ik vond dit lastig te bepalen dus heb ik dit gevraagd aan mijn DEV docent. Ik kreeg als feedback terug dat mijn code niet dynamisch was en dat het handiger was dit eerst dynamisch te maken zodat het een stuk makkelijker wordt de functie modulair te maken. Om het dynamisch te maken zijn getElementById en getElementsByClassName vervangen door querySelectorAll en de variabelen doorgegeven als parameters.

};

Uitvoering 2

Nadat ik de vorige opdracht uitgelegd kreeg door mijn docent heb ik een nieuw stuk modulaire code gemaakt om te laten zien dat ik het nu snap. 


  const changeTextModule = (function () {
    let text = 'Het stuk tekst is nu veranderd (als het goed is)!'
    const textChange = function(){
        let element = document.querySelector('h1');
        element.textContent = text;
    }
    
    return {
        callTextChange: function(){
            textChange();
            console.log(text);
        }
    }
})();

changeTextModule.callTextChange();

Met bovenstaande code kan de gebruiker van mijn code een 'h1' element's value veranderen. Zoals in de vorige modulaire functie, maak ik eerst de module functie aan waarin ik de 'textChange' functie stop zodat het in een private scope staat. Nadat ik mijn functie heb geschreven, return ik een nieuwe functie genaamd 'callTextChange' waarin ik de 'textChange' functie stop. Op deze manier kunnen de gebruikers van mijn code 'textChange' niet veranderen, maar wel gebruiken door de functie 'callTextChange' aan te roepen: changeTextModule.callTextChange();

        Beheerbaarheid


        De student gebruikt Git gedurende het hele traject. De code en Git-project is duidelijk beschreven volgens gangbare richtlijnen in de praktijk.

        Ik probeer dit leerdoel aan te tonen door POC's te maken die beheerbaar en overdraagbaar zijn.




        Git regels


        POC's

        • Er wordt voor elk POC één aparte branch aangemaakt in de repo zodat de repo ‘beheerbaar’ is.
        • Er wordt voor elk POC onderzoek gedaan naar de technologie zodat de POC op de meest efficiënte en relevante manier wordt uitgevoerd.
        • Er wordt voor elk POC één README.md aangemaakt zodat de repo ‘overdraagbaar’ is.


        Website development

        • Er wordt voor elke webpagina één aparte branch aangemaakt in de repo zodat de repo ‘beheerbaar’ is.
        • Er wordt gebruikgemaakt van de GitHub issues functie om taken te verdelen en prioriteiten te stellen. Hierdoor kan er beter samengewerkt worden aan het realiseren van de website.
        • Er wordt een README gemaakt om het project en de ontwikkelde modules uit te leggen.


        README

        Elke README bevat het volgende:


        • Een beschrijving; om uit te leggen wat de gebruiker zal leren uit de POC.
        • Uitleg waarom de POC nuttig is voor het project.
        • Extra essentiële informatie over het onderwerp om de gebruiker van genoeg kennis te voorzien.
        • Een stappenplan, om de gebruiker instructies te geven.
        • Vervolgstappen, om uit te leggen wat de volgende stap is die genomen moet worden om de mogelijkheden verder te verkennen.
        • Bronnen, om de gebruiker te verwijzen naar andere informatiebronnen over het onderwerp.



        Mijn bijdrage


        Organisatie

        Ik heb aangeraden POC's en pagina's/features in aparte branches te ontwikkelen zodat er overzicht blijft en er minder kans is op conflicten. De branches kunnen dan later gemerged worden om de website in elkaar te zetten.

        Verder heb ik advies gegeven voor elk POC een goed gestructureerde POC aan te maken. Nadat mijn README was goedgekeurd door een docent, heb ik mijn README met mijn projectleden gedeeld als voorbeeld om ze op de juiste weg te helpen.

            Ontwikkelomgeving



            De student heeft een onderbouwde (front-end) ontwikkelomgeving samengesteld en laat zien op de hoogte te zijn van de in het werkveld gangbare tools.

            Ik probeer dit leerdoel aan te tonen door een overzicht te geven van de tools die ik gebruik bij het ontwikkelen van webapplicaties.




            Development/programming tools


            Code editor

            Visual Studio Code; ik heb een tijd Brackets geprobeerd, maar vond VScode sneller, betere plugins, extensies en support hebben. Daarbovenop vind ik de UI van VScode aangenamer. Extensies die ik gebruik zijn Live Server, SASS compiler, Beautify en SFTP. Live server gebruik ik om snel te kunnen zien hoe mijn website eruit ziet als ik iets verander. SASS gebruik ik voor gecompliceerde CSS animaties. Beautify gebruik ik voor het netter maken van mijn code (met name HTML). SFTP gebruik ik om meteen al mijn veranderingen in de website code onmiddelijk te uploaden naar mijn live website. Het nadeel is dat afbeeldingen niet mogelijk zijn dus gebruik ik hier FileZilla voor.



            Version control

            GitHub; ik gebruik GitHub als version control systeem, omdat GitHub meer ‘open source’ gericht is. Zo kan ik mijn POC’s makkelijker delen. Nog een reden voor het werken met GitHub is dat ik met GitLab problemen heb met de zichtbaarheid van repo's. GitHub is voor mij dus ook gebruiksvriendelijker.


            GitKraken; ik gebruik GitKraken als Git client omdat het mijn workflow verbetert. GitKraken heeft een goede UI en is goed te gebruiken in groepsprojecten.



            Online tools

            JSON formatter extensions; hiermee kan ik snel een minified of uncompressed versie krijgen van online JSON bestanden (van api’s bijv.)


            VisBug extension; hiermee kan ik net als de Chrome Devtools elementen aanpassen. Ik gebruik VisBug voor het verplaatsen van elementen en veranderen van kleuren, tekst, etc. om snel mijn webapplicaties visueel te verbeteren. Met de Chrome Devtools test ik over het algemeen meer op de functionaliteit van elementen. Ik gebruik deze tools niet alleen om mijn eigen webapplicaties te maken, maar ook om te leren van andere websites hun source code.


            CodePen; als ik wil spelen/experimenteren met wat code gebruik ik vaak CodePen. CodePen is snel, heeft veel support voor frameworks, libraries, SASS en SCSS zodat ik nagenoeg alles kan uittesten.



            Design tools

            Local tools

            Ik kies voor de volgende Adobe programma's, omdat deze programma's goed met elkaar samen werken en dus mijn workflow verbeteren.

            Adobe Photoshop; om afbeeldingen te bewerken voor de webapplicaties.

            Adobe Illustrator; om vector design te maken voor de webapplicaties.

            Adobe XD; om ontwerpen te maken voor de webapplicaties.



            Online tools

            CodePen; ik gebruik CodePen niet alleen voor het testen van Javascript functionaliteit, maar ook voor animaties en vectors. Dit helpt me een interactief design te maken.


            Lorem Picsum; als ik aan het experimenteren ben of een placeholder nodig heb, kan ik snel een afbeelding url krijgen.


            Figma; Ik gebruik Figma als ik met anderen moet samenwerken aan ontwerpen voor apps. Adobe XD is geen live editor en Figma wel.


            Reflectie

            Ik vind dat bovenstaande tools mij enorm geholpen hebben mijn productiviteit en dus efficiëntie te verhogen. Ik denk echter wel dat dit nog kan verbeteren door meer tools te gebruiken. Een tool die ik nog te weinig gebruik naar mijn gevoel zijn de Chrome DevTools. Ik gebruik deze nu vooral voor HTML, CSS en mijn Javascript te debuggen. Ik maak tijdens het debuggen van Javascript nog niet vaak gebruik van event listeners en breakpoints. Ik denk dat als ik dit leer te gebruiken mijn productiviteit en efficiëntie nog hoger kan worden. Mijn doelstelling is daarom ook volgend semester dit leren te gebruiken.
